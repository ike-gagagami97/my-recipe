# Test level policy

PR ごとに「どこまで検証するか」を決めるための方針。  
**レベル定義は合意済み。** どの変更でどのレベルが必須かは、後続のスコア／ルール表で決める（未整備）。

関連: [`workflow.md`](./workflow.md) の④-3。

---

## レベル定義（合意）

積み上げ「全部やれ」ではなく、**ベースライン + 追加ゲート**として組み合わせる。

| Level | 名前 | 環境 | 実行者 | 内容 |
| --- | --- | --- | --- | --- |
| **L0** | 静的検証 | CI / local | エージェント / CI | `lint`、`build`（型チェック含む）、（将来）ユニットテスト |
| **L1** | 動的自動検証 | local（将来は CI も） | エージェント / CI | リポジトリに存在する E2E 等の自動テストを実行（**回帰**） |
| **L2** | エージェント検証 | local | エージェント（Playwright MCP 等） | feature doc の §5 Gherkin / §6 をブラウザ操作で確認 |
| **L3** | Preview エージェント検証 | Vercel Preview | エージェント | L2 相当を Preview URL で再実行（環境・env 込み） |
| **L4** | 人間受け入れ | Preview | 人間（QA） | 手動で受け入れ判断・最終確認 |

### 各レベルの補足

#### L0

- 今あるもの: `npm run lint` / `npm run build`
- ユニットテストが無い変更では unit は N/A（欠けていてもレベル未達にはしない）
- コード変更 PR の **ベースライン候補**（スコア表で確定）

#### L1 と L2 の関係（併存）

| | L1 | L2 |
| --- | --- | --- |
| 何を回すか | 既存の自動テストコード | feature doc の Scenario |
| 向くこと | 回帰 | 新規・テスト未整備・探索的確認 |
| 今のリポ | E2E が無い間は「該当テストがあるときのみ」 | 新規 feature の主戦場 |

どちらか一方に寄せない。揃っている手段を使う。

#### L3（条件付き採用）

- **欲しい**: Preview 固有の問題（env・デプロイ設定）をエージェントで拾うため
- **保留条件**: スコア／ルール表を作るときに「いつ L3 必須か」の判定が難しければ、レベル一覧から外し、必要時だけ人間（L4）または明示依頼にする

#### L4

- **常時必須ではない**
- 文言のみ・振る舞い不変・docs のみなどでは不要になり得る
- 「振る舞いが変わらない」の**一次判定は PR 作成者**（エージェントが下書きしてよい）。人間（QA）は覆せる

---

## 現状の暫定ゲート（スコア表が決まるまで）

スコア／ルール表が無い間:

1. コード変更: 少なくとも **L0**（lint + build）を通す
2. feature 実装: **L2** を推奨（Playwright MCP が使える環境なら）。使えなければその旨を PR に書き、人間が L4 で見るかを判断
3. **L4** は「振る舞いが変わる / 新規画面 / PR作成者が必要と判断」のとき。文言のみ等は省略可
4. Production マージは人間（workflow の D1）
5. エージェントは独自にレベルを拡大解釈しない

---

## 次に決めること（スコア／ルール表）

入力候補（前回の壁打ちから）:

- 画面の新しさ（文言のみ → 既存変更 → 新規）
- データ依存（なし → 読取 → 書込/migration）
- Scenario の数・例外の多さ
- 影響の広さ
- やり直しコスト（migration 等）
- 「振る舞い不変」フラグ（PR作成者）

出力: その PR の必須レベル集合（例: `L0+L2` / `L0+L1+L4`）。L3 を残すかは、ルールの書きやすさで最終判断する。
