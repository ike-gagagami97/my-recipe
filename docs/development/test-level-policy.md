# Test level policy（未整備）

PR ごとに「どこまで検証必須か」を、実装スコープ・難易度・複雑さから **定量的に判定**したい。  
方針は合意前のため、このファイルはプレースホルダとする。

## 現状のゲート（workflow 準拠）

テストレベル表が決まるまでの間:

1. **必須**: feature doc の受け入れ条件を人間（QA）が確認する
2. **推奨**: エージェントは `verify-frontend-change` 等で自己検証する
3. **デプロイ**: Preview 可、Production は人間マージ（D1）

## 将来やりたいこと（メモ）

入力例（スコア化したい軸）:

- 変更ファイル数 / 触る領域（UIのみ、DBあり、認証あり）
- 新規画面の有無
- 外部サービス・migration の有無
- 受け入れ条件の数・例外ケースの数
- PR の依存の複雑さ

出力例:

- L0 lint のみ
- L1 手動受け入れ
- L2 データありの画面確認
- L3 build
- L4 自動E2E
- L5 Preview 必須 など

## 次の検討ステップ

1. 軸の候補を列挙して重みを仮置きする
2. 過去〜今後の数PRで試しにスコアを付けてみる
3. 閾値と必須レベルをこのドキュメントに正式化して `workflow.md` から参照する

決定するまで、エージェントは独自に「L3必須」などと拡大解釈しない。迷ったら feature doc の受け入れ条件に従う。
