## 1. ひとことで言うと（必須）

ユーザーにとって何ができるようになるか。1〜2文。

例: 保存したレシピを一覧で見られる。

## 2. だれの・どんな場面か（必須）

誰の、どんな課題を、どう楽にする機能か。

| 項目 | 書くこと |
| --- | --- |
| だれ | 想定ユーザー（例: 自分用にレシピを溜めたい人） |
| ユーザー課題 | 今困っていること・不便（例: メモアプリに散らばって探せない） |
| 提供価値 | この機能でどう良くなるか（例: 一箇所で一覧でき、すぐ開ける） |
| 使う場面 | いつ・どんなときに使うか（例: 献立を考えるとき、買い物前） |

## 5. 操作の流れ（必須・Gherkin）

ハッピーパスを中心に、[Gherkin](https://cucumber.io/docs/gherkin/reference/) 形式で書く。  
画面名が未定なら「一覧画面」「追加画面」でよい。シナリオは必要数だけ増やす。

```gherkin
Feature: ＜機能名＞
  ＜§1の要約を1行で＞

  Scenario: ＜成功する主な流れの名前＞
    Given ＜前提（誰が・どんな状態か）＞
    When ＜操作＞
    Then ＜画面上でどうなっていてほしいか＞
    And ＜続けて確認したいことがあれば＞
```

例:

```gherkin
Feature: レシピ一覧
  保存したレシピを一覧で見られる

  Scenario: レシピがあるとき一覧にタイトルが出る
    Given レシピが1件保存されている
    When 一覧画面を開く
    Then そのレシピのタイトルが見える

  Scenario: レシピが0件のとき空の案内が出る
    Given レシピが1件もない
    When 一覧画面を開く
    Then まだレシピがないことが分かる案内が見える
```

ルール:

- `Given` / `When` / `Then` / `And` / `But` を使う
- **画面上で観察できること**を書く（内部実装やSQLは書かない）
- 失敗系・例外はここに書いてもよいし、§7 に回してもよい

## 6. 受け入れ条件（必須）

実装完了・レビュー・手動確認の **単一の正本**。  
「確認の観点」もここに含める（別セクションに分けない）。

**検証できる文だけ書く。** 「使いやすい」「いい感じ」は不可。
