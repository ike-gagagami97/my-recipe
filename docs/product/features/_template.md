> **主に QA / プロダクトが書く。** 技術詳細（SQL、ファイル構成、ライブラリ選定）は書かなくてよい。  
> 実装の詳細はエージェントが補う。ここは「できあがりの振る舞い」の契約にする。

## 6. 受け入れ条件（必須）

**検証できる文だけ書く。** 「使いやすい」「いい感じ」は不可。  
QAが Preview またはローカルでチェックする前提。

形式は次のどちらか（混在可）。

### チェックリスト形式

- [ ] …
- [ ] …

### Given / When / Then 形式（任意）

- **Given** … **When** … **Then** …

例:

- [ ] レシピが1件あるとき、一覧を開くとそのタイトルが見える
- [ ] レシピが0件のとき、「まだありません」などの空の案内が見える
- [ ] 必須項目を空のまま保存しようとすると、保存されず入力を促す表示がある

## 7. 空・エラー・例外（ユーザーから見える姿）

技術原因ではなく、**画面上どうなっていてほしいか**を書く。

| 場面 | 期待する見え方・挙動 |
| --- | --- |
| データが0件 | |
| 入力不足・不正 | |
| 通信や保存に失敗 | |
| （その他） | |

分からなければ「未決」と書き、実装前に決める。

## 8. 確認の観点（QAメモ）

手動確認で見たいこと。テストレベル方針が決まるまでのメモ欄。

- 確認環境: ローカル / Preview （どちらで見るか）
- 特に見たい操作:
- 回帰で触っておきたい画面:
- 端末: モバイル幅も見る / デスクトップのみ など

## 9. オープンクエスチョン（人間が Yes/No するもの）

技術詳細ではなく、プロダクト判断が必要なことだけ。

- [ ] 例: 一覧の並びは新しい順でよいか？
- [ ] 例: タイトルは必須でよいか？
