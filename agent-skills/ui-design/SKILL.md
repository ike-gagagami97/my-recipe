# UI design

見た目と情報設計を変えるときの制約です。既存デザインシステムが固まったら、このスキルを更新してそれに合わせる。

## 開始前

1. 現状の `src/app/page.tsx` / `globals.css` / `layout.tsx` を読む
2. 変更が「機能」か「見た目」か切り分ける。機能なら `recipe-feature` も読む

## ハードルール（ランディング・販促面）

- **1 つの構図**: ファーストビューはダッシュボードにしない
- **ブランド優先**: 「My Recipe」がヒーロー級。見出しがブランドを食わない
- **ヒーロー予算**: ブランド + 見出し 1 + 短い文 1 + CTA 群 + 主ビジュアル程度に抑える
- **フルブリード**: ヒーロー画像は端まで。インセットカードやコラージュにしない（既存システムが要求する場合を除く）
- **ヒーロー上にバッジ類を重ねない**
- **カード既定オフ**: 操作の器として必要なときだけカードを使う。ヒーローにカードを置かない
- **1 セクション 1 仕事**
- **動きは 2〜3 の意図的なモーション**（装飾ノイズにしない）

## 避ける見た目

- 紫〜インディゴの定番グラデーション一辺倒
- クリーム地 + テラコッタ + セリフの定番組み合わせ
- 新聞調の細線・ゼロ半径・高密度カラム
- 絵文字の多用、過剰な glow、無意味な `rounded-full` ピルの群れ

## タイポ・背景

- デフォルトの Inter / Roboto / Arial / system だけに頼らない（方針を決めて CSS 変数化）
- 単色ベタ塗りだけにしない（グラデーション・画像・薄いパターンなどで空気感を出す）
- ただし「抽象グラデだけ」を主ビジュアルにしない。料理・プロダクト文脈のアンカーを置く

## 実装メモ

- Tailwind 4 + 既存 `globals.css` のトークンを優先して拡張する
- レスポンシブ（モバイル / デスクトップ）両方でファーストビューが成立するか確認
- アクセシビリティ: コントラスト、フォーカス、意味のある見出し階層

## チェックリスト

- [ ] ナビを除いてもブランドが分かるか
- [ ] ファーストビューに統計・スケジュール・住所などの二次情報が混ざっていないか
- [ ] カードが「操作の器」以外で増えていないか
- [ ] モバイルで崩れないか
